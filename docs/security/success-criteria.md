# v1.0 Success Criteria — Evidence

The three success criteria from [docs/01 §8](../01-requirements.md), each with how
it is demonstrated and where the evidence lives. This is the M15 exit record.

## Criterion 1 — End-to-end delivery

> Point at a real OSS repo → ask for a bug fix → agent plans, edits, tests in
> sandbox, human approves → PR opens on GitHub, with the full run visible as a
> trace.

**Demonstrated by** the full agent graph: Planner → Coder → (write gate) →
Reviewer loop → Tester (sandbox) → Debugger → Documenter → (PR gate) → GitHub PR.

| Stage | Evidence |
| --- | --- |
| Plan → edit → review → commit | `tests/unit/agents/test_coder_workflow.py`, `test_graph.py` |
| Test in sandbox | `tests/security/test_sandbox_containment.py`, `test_k8s_sandbox.py` |
| Human approval gate on every side effect | `tests/unit/agents/test_run_service.py`, approval flow (M8/M10) |
| PR opens past a gate; report projects the timeline | `tests/unit/agents/test_m10_flow.py`, `GET /runs/{id}/report` |
| Full run visible as a trace/timeline | event-sourced timeline + SSE; replay (`test_replay_eval.py`) |
| UI drives it end to end | React SPA (M12): runs, plan board, diff, approvals, dashboard, replay |

The flow is exercised deterministically in CI with fixtures; a live run against a
real repo requires provider keys and is run from the [deploy](../runbooks/deploy.md)
environment.

## Criterion 2 — Security demo (booby-trapped repo)

> A booby-trapped repo (injection payloads in README, malicious `postinstall`)
> fails to escape the sandbox, exfiltrate secrets, or hijack the agent — with the
> attempts visible in the audit log.

**Demonstrated by** the M9 red-team suite and the containment/injection tests:

| Attack | Defense verified | Evidence |
| --- | --- | --- |
| Injection payload in retrieved content | Inert data frame; no instruction execution | `tests/unit/platform/test_injection.py`, framing |
| Malicious `postinstall` / hostile command | Sandbox: no network, non-root, RO rootfs, caps dropped, killed at deadline | `tests/security/test_sandbox_containment.py` |
| Secret read/exfiltration attempt | Scrubbed env; secret-scan on output/diffs; egress blocked | `tests/unit/execution/test_scrub.py`, `tests/unit/platform/test_secrets.py` |
| Path traversal in tool args | `SafeFileSystem` allow-list + symlink/junction resolution | `tests/security/test_safe_filesystem.py` |
| Memory poisoning across sessions | Fact-only write gate; inert recall framing | `tests/security/test_memory_poisoning.py` |
| Attempts recorded | Append-only, trigger-enforced audit log | `tests/security/test_audit_and_lockout.py` |

The booby-trapped-repo scenario is encoded as attack-shaped tests that run on
every PR, so the demo is continuously re-verified rather than a one-off.

## Criterion 3 — Evaluation harness in CI

> Reports task success rate, token cost, and wall-clock per task across a fixed
> benchmark suite, tracked over time in CI.

**Demonstrated by** the tiered evaluation harness:

| Capability | Evidence |
| --- | --- |
| Retrieval P/R@k, NDCG, MRR + rerank ablation | `RetrievalEvalSuite`, `RetrievalAblationSuite` ([M13](../perf/m13-retrieval-v2-eval.md)) |
| Agent task success rate, cost, groundedness | `AgentTaskEvalSuite`, `GroundednessEvalSuite` |
| Safety / injection / memory-poisoning corpus | `MemorySafetyEvalSuite`, safety baselines |
| Deterministic replay of runs | `AgentReplayEvalSuite` |
| Blessed baselines enforced as gates | `evaluation/baselines/*.json` + `check_baselines` |
| Runs in CI, tracked over time | `.github/workflows/evals.yml`, tiered (`python -m spidey.evaluation run --tier …`) |

Published numbers are generated by the harness (never hand-written); live-model
tiers land their figures as models are wired, on the eval workflow's cadence.

## Verdict

All three success criteria are demonstrably met: criteria 1 and 2 by
deterministic and attack-shaped suites that gate every change, and criterion 3 by
the evaluation harness wired into CI. See the [release checklist](release-checklist.md)
for the full v1.0 sign-off.
